Headings

use # followed by a space. six levels, h1 through h6. always one space between the hash and the text or it won't parse right.

markdown 
# Heading 1
## Heading 2
### Heading 3
#### Heading 4
##### Heading 5
###### Heading 6

Heading 1
=========

Heading 2
---------
heads up: h1 should appear once per doc, as the title. GitHub auto-generates anchor links from headings.

Emphasis

asterisks work better than underscores across parsers. underscores inside words like some_var_name won't trigger italic in CommonMark.

markdown 
*italic*
_italic_

**bold**
__bold__

***bold and italic***

**bold with _nested italic_**

Fenced Code Blocks

triple backticks open and close the block. put the language name right after the opening backticks for syntax highlighting.

markdown

```python
    print("hello world")
```

```javascript

```

```bash

cd repo && npm install
```

```json

  "name": "project",
  "version": "1.0.0"
```



Tables

GFM extension. the separator row is required. outer pipes are optional but look cleaner.

markdown 
| col 1 | col 2 | col 3 |
| ----- | ----- | ----- |
| cell  | cell  | cell  |
| cell  | cell  | cell  |

| left         | center        | right         |
| :----------- | :-----------: | ------------: |
| :---  left   | :---:  center |   ---:  right |
note: tables can't contain block elements like code blocks or lists.

Mermaid Diagrams

GitHub renders these natively. use the mermaid language tag on any fenced block.

flowchart

mermaid 
```mermaid
flowchart TD
    A[Start] --> B{Decision}
    B -->|Yes| C[Do this]
    B -->|No| D[Do that]
    C --> E[End]
    D --> E
```

sequence diagram

mermaid 
```mermaid
sequenceDiagram
    participant Client
    participant Server
    Client->>Server: HTTP Request
    Server-->>Client: HTTP Response
```

gantt

mermaid 
```mermaid
gantt
    title project timeline
    dateFormat YYYY-MM-DD
    section phase 1
    task one   :a1, 2024-01-01, 30d
    task two   :after a1, 20d
```

er diagram

mermaid 
```mermaid
erDiagram
    USER {
        int id
        string name
    }
    POST {
        int id
        int user_id
    }
    USER ||--o{ POST : writes
```

README Structure

the readme is the front page of your project. put the important stuff first, use code blocks for every command, link the license instead of pasting it.

markdown 
# project name

one or two sentences about what this does.

## features

- feature one
- feature two

## requirements

- node.js 18+
- npm 9+

## installation

```bash
git clone https://github.com/user/project.git
cd project
npm install
```

## usage

```bash
npm start
```

## configuration

| variable       | default | description      |
| -------------- | ------- | ---------------- |
| `PORT`         | `3000`  | server port      |
| `DATABASE_URL` | —       | connection string |

## contributing

1. fork the repo
2. `git checkout -b feature/name`
3. `git commit -m "add feature"`
4. `git push origin feature/name`
5. open a pull request

## license

[MIT](LICENSE)
